# 在庫管理 STEP2 総まとめと次ロードマップ

## 1. STEP2の目的

STEP2は、在庫管理を「画面デモ中心の擬似運用」から「実データ中心の業務運用」へ移行するためのフェーズです。主目的は次の5点です。

- mockデモから実データ運用へ移行する。
- 在庫数量を履歴（movement）で追えるようにする。
- 入庫予定/発送予定をDBで管理し、予定と実績を切り分ける。
- パチマート出品/取引（Exhibit/Dealing）と在庫管理を接続する。
- 見込み利益表示の入口を実装し、次段階の利益管理へつなぐ。

---

## 2. 実装済み機能

### 2-1. 在庫管理
- `InventoryItem` を在庫の実体として管理。
- `InventoryMovement` を数量履歴として管理。
- 在庫一覧の実装。
- 在庫詳細の実装。
- 在庫登録の実装。
- 在庫編集の実装。
- 数量変更時に `InventoryMovement` を作成する運用に統一。

### 2-2. 入庫管理
- `InboundSchedule` を導入し、入庫予定をDB管理。
- 入庫予定一覧を実装。
- 入庫予定登録を実装。
- 入庫予定編集を実装。
- 取消（キャンセル）を実装。
- 入庫完了を実装。
- 入庫完了取消を実装。
- `sourceType` / `sourceId` / `dedupeKey` を実装し、起票元追跡と重複抑止を可能化。
- 購入 `Dealing` からの入庫予定自動作成を実装。
- 入庫先（保管場所）必須化を実装。
- 入庫先未設定フィルタを実装。
- 既定倉庫補完（デフォルト値の補完）を実装。

### 2-3. 発送管理
- `OutboundSchedule` を導入し、発送予定をDB管理。
- 発送予定一覧を実装。
- 発送予定登録を実装。
- 発送予定編集を実装。
- 取消（キャンセル）を実装。
- 発送完了を実装。
- 発送完了取消を実装。
- `sourceType` / `sourceId` / `dedupeKey` を実装し、起票元追跡と重複抑止を可能化。
- 成約 `Dealing` からの発送予定自動作成を実装。

### 2-4. 出品/取引連携
- `InventoryExternalLink` を実装し、在庫と外部ドメインを接続。
- 在庫詳細から出品フォームへ初期値引き継ぎを実装。
- 出品作成後に `InventoryExternalLink` を作成する連携を実装。
- `Exhibit.status` → `InventoryItem.listingStatus` の同期を実装。
- Exhibit status更新serviceの共通化を実装。
- `Dealing` から発送/入庫予定を自動作成する導線を実装。

### 2-5. 利益表示
- `purchaseUnitPrice` / `plannedSaleUnitPrice` を在庫に保持。
- 見込み粗利表示を実装。
- `calculateProjectedProfit` を実装。
- `InventoryProfitSummary` / `InventoryProfitMini` を実装。
- profit関連の単体テストを実装。

## 3. 重要な設計思想

STEP2の実装は、以下の思想を一貫させています。

1. 在庫数量は `InventoryMovement` で履歴管理する。  
2. movementは削除しない（監査可能性を維持）。  
3. 取消は「reverse movement（逆仕訳）」で表現する。  
4. 成約/購入時点では在庫数を直接動かさない。  
5. 入庫完了/発送完了時に初めて在庫数量へ反映する。  
6. 販売管理（取引・金額）と倉庫管理（数量・場所）を分ける。  
7. Exhibit/Dealing/Naviとは `InventoryExternalLink` と `source*` 系項目で疎結合に接続する。  
8. `dedupeKey` で二重反映・二重起票を防ぐ。  
9. 原価/利益は数量管理と分離し、段階導入する。  

---

## 4. 現在の主要導線（デモ推奨順）

デモ時は、次の順に見せると「予定→実績→在庫反映→利益入口」の全体像を伝えやすいです。

1. `/inventory`（在庫管理トップ）
2. `/inventory/items`（在庫一覧）
3. `/inventory/items/[id]`（在庫詳細）
4. 在庫詳細から出品フォームへ遷移（初期値引き継ぎ）
5. 出品作成後、在庫とのリンク表示（`InventoryExternalLink`）
6. 成約発生をトリガーに発送予定作成（自動起票）
7. `/inventory/outbound`（発送予定一覧）
8. 発送完了を実行し在庫数量が減ることを確認
9. 購入発生をトリガーに入庫予定作成（自動起票）
10. `/inventory/inbound`（入庫予定一覧）
11. 既定倉庫補完・入庫先確認（未設定フィルタ含む）
12. 入庫完了を実行し在庫数量が増えることを確認
13. 在庫詳細/一覧で見込み粗利表示を確認

## 7. 最終推奨（次に実装するなら）

**推奨はご希望どおり `STEP2-K` から開始**です。理由は次のとおりです。

1. **既存実装価値を最短で可視化できる**  
   すでにある在庫・入庫・発送・連携・見込み粗利の機能を、1本の業務ストーリーとして示せるため。

2. **非破壊で進めやすい**  
   大きなschema追加や会計設計の確定を待たずに着手でき、リスクが低い。

3. **次STEPの要件確定に効く**  
   ダッシュボード実運用で「本当に必要な伝票項目/KPI」が見え、STEP2-L（伝票最小モデル）の精度が上がる。

4. **ステークホルダー合意が取りやすい**  
   数量導線と利益入口が同じ画面群で確認でき、優先順位判断がしやすい。

### 推奨実行順（具体）
- まずSTEP2-Kで導線・KPI可視化を完成。
- 次にSTEP2-Lで `PurchaseRecord` / `SalesRecord` の最小導入。
- その後STEP2-M/N（CSV・QR/棚卸）で運用入力を効率化。
- 最後にSTEP3で実粗利・台帳突合・分析へ拡張。
