# STEP2-O: 在庫ダッシュボード/履歴の総合確認ドキュメント

## 1. 対象範囲
本ドキュメントは、STEP2-K〜Nで実装した在庫ダッシュボード/履歴の回帰確認観点を固定化することを目的とする。

- `/inventory`
- `/inventory/activity`
- KPIカード
- 最近の動き
- イベントフィード
- フィルタ
- ページング

---

## 2. `/inventory` KPI確認観点
確認対象KPI:
- 在庫件数
- 在庫台数
- 入庫予定件数
- 発送予定件数
- 入庫先未設定件数
- 見込み粗利合計

### 2.1 在庫件数
- 集計対象
  - 在庫管理対象として扱う在庫レコードの件数。
- 除外条件
  - 論理的に無効化されたレコード、在庫管理対象外データ（運用で除外対象と定義したもの）。
- 期待表示
  - KPIカードに件数として整数表示される。
- 注意点
  - 在庫台数（数量合計）とは意味が異なるため、件数と数量を混同しない。

### 2.2 在庫台数
- 集計対象
  - 在庫数量（on hand）を全件合算した値。
- 除外条件
  - 集計対象外在庫、無効データ。
- 期待表示
  - KPIカードに数量合計として表示される。
- 注意点
  - 在庫件数とは異なり、1レコードに複数台数が含まれる。

### 2.3 入庫予定件数
- 集計対象
  - 未完了の入庫予定（これから入庫される予定）の件数。
- 除外条件
  - 取消済み、完了済みの入庫予定。
- 期待表示
  - KPIカードに最新状態の件数が表示される。
- 注意点
  - 入庫完了/取消操作後に件数が即時整合していることを確認する。

### 2.4 発送予定件数
- 集計対象
  - 未完了の発送予定（これから発送される予定）の件数。
- 除外条件
  - 取消済み、完了済みの発送予定。
- 期待表示
  - KPIカードに件数が表示される。
- 注意点
  - 発送完了/取消後に件数反映が正しいこと。

### 2.5 入庫先未設定件数
- 集計対象
  - 入庫予定のうち、入庫先（倉庫/ロケーション等）が未設定の件数。
- 除外条件
  - 入庫先が設定済みの予定、取消済み/完了済み予定。
- 期待表示
  - KPIカードで要対応件数として可視化される。
- 注意点
  - 運用上の入力漏れ検知KPIのため、設定有無の判定基準（null/空値）を明確に扱う。

### 2.6 見込み粗利合計
- 集計対象
  - 対象在庫（または予定）に紐づく見込み粗利の合計。
- 除外条件
  - 粗利計算不能データ、無効データ、対象外ステータス。
- 期待表示
  - KPIカードに通貨フォーマット（桁区切り）で表示される。
- 注意点
  - 粗利は売上/原価の定義に依存するため、算出元の前提（未確定値を含むか）を固定して確認する。

## 3. `/inventory` 最近の動き確認観点
確認対象イベント:
- 入庫予定作成
- 発送予定作成
- 入庫完了
- 発送完了
- 入庫完了取消
- 発送完了取消
- 手動調整
- 予定取消

確認内容:
- `occurredAt desc`
  - 新しいイベントが上に並ぶこと。
- バッジ表示
  - イベント種別ごとに期待するバッジが表示されること。
- 数量差分
  - 増減が符号付きで読み取れること（例: `+5`, `-3`）。
- 詳細リンク
  - 対応する詳細画面へ遷移できること。
- 直近10件
  - 最新10件に制限され、11件目以降は表示されないこと。

---

## 4. `/inventory/activity` 確認観点
- 最大50件/ページ
  - 1ページあたりの表示件数上限が50件であること。
- `page` query
  - `?page=1`, `?page=2` でページ遷移が機能すること。
- `type` filter
  - 種別指定でイベントが絞り込まれること。
- `range` filter
  - 期間指定で対象期間のイベントのみ表示されること。
- フィルタ変更時 `page=1`
  - 種別/期間変更でページが1へリセットされること。
- 不正pageのfallback
  - `page=0`, 負数, 過大値, 数値以外指定時に安全な値へフォールバックすること。
- 件数表示
  - 総件数または現在表示件数が仕様通り表示されること。
- `occurredAt desc`
  - 並び順が常に新しい順であること。

---

## 5. 手動確認手順
1. `/inventory` を開く。
2. KPI（6指標）の値と表示形式を確認する。
3. 最近の動きで最新イベント順・バッジ・数量差分を確認する。
4. 最近の動きから「すべて見る」を押下する。
5. `/inventory/activity` へ遷移したことを確認する。
6. 種別フィルタ（type）を変更し、結果が絞り込まれることを確認する。
7. 期間フィルタ（range）を変更し、対象期間のみ表示されることを確認する。
8. `page=2` へ遷移し、ページングが動作することを確認する。
9. `page=0` や `page=abc` など不正pageを試し、fallbackを確認する。
10. 一連確認後、再度 `/inventory` に戻って主要KPIとの整合を確認する。

## 6. 期待されるDB変化との関係
- 入庫完了
  - `InventoryMovement` に `INBOUND` が作成されること。
- 発送完了
  - `InventoryMovement` に `OUTBOUND` が作成されること。
- 入庫/発送取消
  - 取消時は `ADJUSTMENT`（reverse）相当の差分で整合が戻ること。
- Dealing由来予定
  - `sourceType` / `sourceId` / `dedupeKey` により起点と重複防止が追跡できること。
- activityのデータ構成
  - activityは新テーブルを追加せず、既存データから正規化して表示していること。
